# Module 2 - Generative Industrial Digital Twin

This Week 2 module implements the work assigned to Person 1 and Person 2. It consumes the CSV telemetry generated by `Module 1 - Data Acquisition` and provides validated state management, online behaviour learning, CSV-based real-time synchronization, and scenario simulation.

## Responsibility map

- Person 1: `production_line_a/`, `production_line_b/`, and `boiler/`
- Person 2: `solar/`, `battery/`, and `grid/`
- Person 3 directories (`hvac/`, `compressor/`, `common/`, and integration work) are intentionally not included in this delivery.

Each delivered system contains the five files required by the Week 2 brief: `digital_twin.py`, `state_manager.py`, `behavior_learning.py`, `realtime_sync.py`, and `simulation.py`.

The requested research topics and design rationale are documented in [`RESEARCH.md`](RESEARCH.md).

## Running the tests

From the repository root, run:

```bash
python3 -m unittest discover -s module2_digital_twin/tests -v
```

The tests synchronize every twin with its corresponding Week 1 dataset and verify state validation, behaviour updates, and scenario output.

## Basic usage

```python
from module2_digital_twin.solar import SolarDigitalTwin

twin = SolarDigitalTwin()
twin.realtime_sync.sync_csv(
    "Module 1 - Data Acquisition/outputs/solar_plant.csv"
)
result = twin.simulation.run({"irradiance_w_m2": 900})
print(result["predicted_metrics"])
```

`sync_csv(path, only_new=True)` skips source timestamps already processed. `simulation.run(overrides)` creates a what-if result without changing the synchronized live state. Results can be written to the Week 2 outputs area with `simulation.save(result, path)`.

## Output folders

The brief's output locations are present under `outputs/`: `digital_states/`, `simulation_results/`, `logs/`, and `reports/`. Runtime output is written only when `simulation.save` is called, keeping tests repeatable.
